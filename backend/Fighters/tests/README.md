# Tests

## Перед началом

Для выполнения данного задания вам необходимо, чтобы предыдущая лабораторная работа **Fighters** была полностью завершена и принята. Это важно, так как все тесты в данном задании должны покрывать уже готовую, рабочую реализацию. Если задание Fighters не сдано, выполнение тестирования теряет смысл: либо за основу будет взята неактуальная логика, либо вам придется постоянно возвращаться и менять тесты при доработке лабораторной.

## Задача

Ваша задача - написать unit-тесты для классов из лабораторной работы **Fighters**. Для этого используйте:
    **XUnit** в качестве тестового фреймворка (<https://xunit.net/> )
    **Moq** для создания моков (<https://github.com/moq/moq4> )

Тесты должны максимально полно покрывать возможные сценарии использования ваших классов («позитивные» и «негативные» случаи, пограничные значения и др.).

## Требования и рекомендации по написанию unit-тестов 

### 1. Структура (иерархия) тестового проекта

Проект с тестами должен повторять структуру основного проекта Fighters. Если в вашем проекте есть папки/areas (например, «Services», «Models», «Controllers»), в тестовом проекте создайте аналогичную структуру.

Классы тестов должны лежать в соответствующих папках (например, все тесты для класса `FighterService` - в папке `Services\FighterServiceTests` ).

### 2. Наименование тестовых классов и методов

- **Тестовые классы** называйте как `<ИмяТестируемогоКласса>Tests`.
Пример:
  - Класс: `FighterService` → Тестовый класс: `FighterServiceTests`

- Если вы тестируете отдельные методы класса, создайте отдельные методы внутри `FighterServiceTests` . Если методов очень много - можете вынести тесты для каждого из них в отдельные файлы или подпапки внутри `FighterServiceTests` .

- **Тестовые методы** именуйте по стилю Microsoft:
`<НазваниеМетода_Условия_ОжидаемоеПоведение>`

Примеры:
- `Attack_TargetIsAlive_DealsCorrectDamage()`
- `Constructor_InvalidArguments_ThrowsException()`
- `Heal_WhenHealthy_NoChange()`

Такой стиль облегчает понимание, что именно и при каких условиях проверяет тест. Подробнее - [link](https://learn.microsoft.com/en-us/dotnet/core/testing/unit-testing-best-practices#follow-test-naming-standards).

### 3. Использование Moq

Все зависимости классов (например, сервисы, репозитории или внешние системы), которые внедряются через конструктор или через интерфейсы, должны быть замокированы с помощью Moq.

Не пишите руками «фейковые» реализации классов для тестов - используйте возможности Moq для задания поведения, проверки вызова методов, возврата нужных значений и т.д. Разберитесь с базовыми конструкциями Moq:

`Mock<T>()` - создание мока

`Setup()` - задание поведения

`Verify()` - проверка количества или факта вызовов

### 4. Генерация и переиспользование тестовых данных

Не копируйте и не дублируйте логику создания тестовых данных (например, объекты Fighters, mock-объекты и пр.) внутри каждого теста.

Используйте приватные методы или фабрики в рамках тестового класса для создания необходимых экземпляров.

Если набор тестовых данных универсален и нужен многим тестам - вынесите его в статические методы или поля.

Не стесняйтесь использовать библиотеки типа NBuilder для генерации сложных объектов.

### 5. Рефакторинг тестируемого кода (по необходимости)

Если вы видите, что некоторая часть бизнес-логики написана так, что её неудобно и/или невозможно покрыть качественными unit-тестами (например, жёстко зашитые зависимости, нет инъекции внешних сервисов, ветвление кода без тестируемых условий и т. п.), то вы должны провести рефакторинг боевого (рабочего) кода для улучшения тестируемости.

## Часто задаваемые вопросы

**1\. Какие классы нужно тестировать?**

Тестируйте все классы из вашей лабораторной работы, содержащие бизнес-логику (обычно это сервисы, утилиты, модели с логикой). Не тестируйте «глупые» DTO или модели данных, если в них нет логики.

**2\. Обязательно ли использовать и XUnit, и Moq?**

Да, в этом задании мы осваиваем и фреймворк тестирования, и библиотеку моков.

**3\. Если в классе нет зависимостей, нужно ли использовать Moq?**

Нет, используйте моки только там, где тестируемый класс зависит от других интерфейсов/ сервисов.

**4\. Надо ли писать тесты на приватные методы?**

Нет, unit-тесты проверяют публичное поведение класса. Приватные методы тестируются опосредованно через публичные.

**5\. Какой критерий успешности? Сколько тестов писать?**

Покрытие тестами должно быть достаточным: вся важная логика, все ветки, все случае должны быть проверены. Минимальное требование - «зелёный» основной тестовый прогон и отсутствие явных пропусков.

**6\. Могу ли я менять архитектуру боевого кода ради тестируемости?** Да, и это правильно. Если приходится менять параметры конструктора, внедрять новые интерфейсы/абстракции - делайте это.

**7\. Как запустить тесты?**

Используйте встроенные средства вашей IDE (например, Visual Studio Test Explorer), либо команду dotnet test в консоли.
